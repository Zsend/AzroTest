AZRO Treasury Standard v1

A policy-first standard for Bitcoin treasury programs designed to be board-approvable: durable through full cycles, measured in BTC terms, and governed by clear controls and monthly reporting.

Purpose

AZRO Systems is a rules-based treasury operating system designed to convert recurring operating surplus into long-term Bitcoin reserves while preserving an operations-first cash buffer and maintaining board-ready governance.

This standard documents the minimum elements required for a disciplined, auditable program that can survive full market cycles without forced actions.

Principles

  • No leverage. No margin, borrowing, derivatives, lending, rehypothecation, or yield products.
  • Operations first. Maintain a cash buffer floor sized to avoid forced selling.
  • Rules over emotion. Written policy + repeatable execution + documented exceptions.
  • Separation of duties. Approver ≠ executor; multi-party controls for meaningful balances.
  • Audit-ready reporting. Monthly BTC, BTC-eq, buffer runway, baseline comparison, and exception log.

Definitions (metrics standard)

These definitions are used across AZRO reporting so boards and finance teams can review the same KPI set every month.

Metric Definition
BTC balance BTC held in custody (retain statements/addresses).
BTC-eq BTC + (remaining cash buffer ÷ spot BTC). A comparable reserve KPI.
Buffer runway Cash buffer ÷ monthly operating expenses (optional, board-friendly).
Baseline Auto-buy using the same dollars and schedule — the control group.
Sats accretion (Strategy BTC − baseline BTC) measured in satoshis; compounding advantage.
Exceptions Any trim, custody change, incident, or policy override (logged with rationale + approvals).

Governance & controls

A Bitcoin treasury program is a treasury function — it must be governed like one. Minimum controls:

  • Written policy memo (budget sizing, buffer floor/target, mode, custody, trim limits, cadence, non-negotiables).
  • Defined roles (Owner/Board approves; Treasury lead runs process; Executor executes; Control owner reviews).
  • Separation of duties (approver ≠ executor; two-person approvals recommended for custody transfers).
  • Artifacts retained (confirmations, statements, addresses, exception logs, monthly reports).
  • Exception handling (trigger, rationale, approvals, execution evidence, post-action review).

Controls map

Approvals, execution, and reporting are separated to reduce key-person and “stress decision” risk.

For meaningful balances, adopt multi-party controls and retain evidence (addresses/confirmations) alongside monthly reporting.

Operating cadence

Weekly
Allocate per policy → execute buys → retain confirmations.
Monthly
Publish KPI report → baseline comparison → exception log → sign off.
Quarterly
Reaffirm policy sizing (budget + buffer) and roles; update controls if needed.

Reporting minimums

Each month, produce a consistent report that includes at least:

  • Balances (BTC balance with custody evidence).
  • Reserve KPIs (BTC-eq + sats accretion vs baseline).
  • Liquidity (buffer vs target and/or buffer runway).
  • Execution log (buys/sells, fees, confirmations retained).
  • Exceptions (trims, custody changes, incidents, policy overrides — even if “none”).

Implementation phases

Disciplined posture: start small → review monthly → scale only when stable.

1
Setup
Runway context → buffer policy → weekly budget → mode → custody & reporting.
2
Policy
Write policy memo; define roles, approvals, and non-negotiables.
3
Pilot
Run 8–12 weeks; validate operations + reporting before scaling.
4
Review
Monthly review; correct process issues; maintain buffer discipline.
5
Scale
Increase budget only when stable and controls are proven.
